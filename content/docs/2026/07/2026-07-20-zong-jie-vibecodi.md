[体验过Codex](https://mp.weixin.qq.com/s?__biz=MjM5ODczOTMzMA==&mid=2247487790&idx=1&sn=d3d3399a57b1929ddf71ffbf777b9fd9&scene=21#wechat_redirect)，再结合Cursor主推的Agents Window，我感觉未来的编码趋势都只在指令传达的是否准确，以及Skill选择的是否合理，似乎写代码不看代码是真真可行的。

于是，我的主要编码工具在最近的一个季度，已然从 Cursor 转变为 Codex。

三个月前，我又一次参与到一个从零开始的全新项目，所负责的部分依然被称作“Agent”，功能简单些说，是基于某个 Agent 框架以某种较固定流程调用模型。当时这项目要得急，组内负责进度同事常挂嘴边的一句话是：“你们不要管那些可复用可迭代可掌控，先直接让 AI 把功能做出来再说。”

我们最初的开发范式，是产品直接让 AI 基于相似产品输出产品文档与原型，再由程序员让 AI 基于这产品文档与原型，结合 Agent 框架的源码与文档生成最终功能。

真的，第一版出来的产品像模像样的，该有的交互该有的模块该有的风格，都能对上（开发过程中，我们的前端写了许多Skill，我们的后端也写了很多的代码约束）。但当这产品真正交给用户试用之后，问题全部暴露出来：某个功能没有实现，某条限制没有写全，某个表单切换条件后不能点……

总之是，Bug 不断，而当问起 Bug 来源于何处，我并不能很快给出答案，只这样回复：“你等我问一下 Codex 先。”我不知道模型为什么会生成这样的结果，我不确定约束规则到底如何填充，“用户”反馈、我的直觉都对我说：“这系统中，坑多着呢。”

我将这阶段（时长一个月多一点）的产出称之为“空中楼阁”，因为看着 AI 写的这许多代码，我毫无底气。

此时，新进来一位产品，他的工作是基于我们系统搭建一个真正给目标用户使用的 Agent。系统难用，他的搭建过程不很顺利，于是他基于自己的理解——用户需求、Agent 框架——对搭建模块做了大刀阔斧的改造（他拥有改造权限），这次改造的核心，是模块中所有上架的功能，能砍则砍，只留下我们人类自己能够看懂能够掌控的部分。

此后时间又过去一个月多一点，我们借助 AI 对系统再做一次完善。（当然，前面阶段所做工作并非没有用处，过程中也一直在做相关知识的积攒，后期对想要功能已经有一个能够复述的框架性认知，此时的再开发，也就变得有把握许多。）

现阶段，虽然新的功能依然完全由 AI 添加，对新添加功能的理解依然不够全面，但过去已经验证可用的那些，是已经多出许多掌控感的。

以上，是二季度工作的一个简单总结，它再精简为一句话是：Codex 是性能强劲的一辆豪车，作为司机的我开着它在一个完全陌生的国度摸索，它不知道目标，我也说不清方向。

我反思这开车过程，大概想出这样几条改进线路：

做驾驶员的自己，一定要知道前进方向是怎样的，哪里转弯哪里减速，也必须得心里有谱。[Codex（AI）真的很强](https://mp.weixin.qq.com/s?__biz=MjM5ODczOTMzMA==&mid=2247488017&idx=1&sn=3a0a430e5c3bc6a59f27fedced8d0f86&scene=21#wechat_redirect)，但什么都不懂只让 AI 往前开，是带些敷衍的行事方式。Codex 很快，基于 Codex 的开发也很快，“我已经搞定一个版本”有一定概率只是说来让别人“安心”让别人误解的。真实情况如何？可能是不理解需求的盲目乐观，也可能是内心没底的硬撑头皮。

于是 Codex 时代，“慢工出细活”依然有效，开始之前的多想想，将需求想得更清楚一些，多看看，对所用新框架多一些了解，代码写就之后自己真人——毕竟系统的用户是人类而非 AI——的多测测，是依然必要且必不可少的。

代码是依然需要人工 Review 的，虽然 Cursor 时代已经确认 AI 写的代码很是不错，但让它无休止膨胀（即便明确告知规范、有约束，每次提交之前都走过一遍 Review 流程，功能多了之后，依然会膨胀）下去，依然能够看到些坏味道：单文件太长、文件平铺太多不模块化、职责混杂，甚至重复实现等等等等。

这个季度，我已经开始让 Codex 帮着写前端代码，相较[三年半之前试学前端](https://mp.weixin.qq.com/s?__biz=MjM5ODczOTMzMA==&mid=2247485309&idx=1&sn=7867fd064c1e01e072cf3b56052bfa40&scene=21#wechat_redirect)时的跌跌撞撞，现在的瓶颈已经明显变成如何准确表达自己需求，以及如何设计才能让 UI 好看。所以 Codex 时代需要改进的点，依然是过去的老生常谈：持续学习以增加自己知识的广度。

（另一个更具体示例，是自从[三月将自己网站基于 Hugo 与 GitHub 重新部署](https://mp.weixin.qq.com/s?__biz=MjM5ODczOTMzMA==&mid=2247487618&idx=1&sn=2440b19d7fb9e03af820595e93ab077f&scene=21#wechat_redirect)以来，这个季度有对它进行持续的调整，过程中，我明显感受到自己于审美层面的表达不畅，于是将《设计心理学》加入书架，目前阅读进度不多，但已经很是喜爱这本书了。）

五月中旬，在我最是繁忙那段时间（月累计工时又超过 250 小时），在我刚刚完全接受 Codex 当时，我有好奇一些前同事当前的编码方式，于是咨询好几位前同事，得到这样几条小样本且可能已经过时（世界真的变化很快）的信息。

腾讯的游戏项目组，他们面向用户的项目代码依然古法编程由人工写就，但工具代码全交给 AI 处理。腾讯给每个程序员的月 Token 额度是 1000 刀。

网易大概有一个自研的类似 Sub2API 的中转平台，大家的 Token 都走这平台过，对用户的使用有各种维度的统计，他们的月度 Token 额度，是人民币 5000 元。

而字节因为有自己的 Trae，所以内部只能用它。

整理二季度总结过程中，我以为它的字数太多，于是将至少两个月前生出的“整理空中楼阁”想法放大成为单独的本篇。

AI 真的很强，但我的自我迭代，可绝不能停。 <small>（<a href="https://mp.weixin.qq.com/s/zMxK-GV3ZaPKdjgI-6kr-w" rel="noopener noreferrer">原文链接</a>，更新于2026-07-20。）</small>
